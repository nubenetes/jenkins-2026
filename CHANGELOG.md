# Changelog

All notable changes to **jenkins-2026** are documented here, following
[Keep a Changelog](https://keepachangelog.com/en/1.1.0/) and
[Semantic Versioning](https://semver.org/spec/v2.0.0.html).

- **`Unreleased`** — changes accumulate under the section below as PRs merge; cutting
  a release renames it to a dated `## [vX.Y.Z]` section and starts a fresh `Unreleased`.
- **Versioning** — **`v1.0.0` is the stable baseline.** From 1.0 on: **minor** (`v1.Y.0`)
  at a feature milestone, **patch** (`v1.Y.Z`) for hotfixes, **major** (`v2.0.0`) for
  breaking changes. Every version is a git **tag** *and* a GitHub **release**, one-to-one,
  cut from its section by [`scripts/cut-release.sh`](scripts/cut-release.sh). See
  [`RELEASING.md`](RELEASING.md) for the full flow.
- **Pre-1.0 history** — the **`v0.x` line (v0.1.0 – v0.29.0)** was rapid pre-release
  development; those releases are a **frozen historical record** (never renumbered — that
  would break version immutability). Versions **≤ v0.28.56** live verbatim in
  [`CHANGELOG-ARCHIVE.md`](CHANGELOG-ARCHIVE.md) to keep this file scannable; the
  [release index](#release-index) links every milestone.

## [v1.6.1] - 2026-07-19

_Corrects the Git LFS troubleshooting entry with this organisation's real quota and where the bandwidth actually goes._

### Documentation
- **docs/902 § "exceeded its LFS budget" now cites the real allowance instead of the free tier** ([`docs/902-TROUBLESHOOTING.md`](docs/902-TROUBLESHOOTING.md)): the entry added in v1.6.0 explained the failure using GitHub's **free** tier (1 GB storage + 1 GB bandwidth), which is true in general but wrong for this repo and made it read as though the platform were sitting near a cap. The `nubenetes` org actually has **10 GB + 10 GB**, and measuring it on 2026-07-19 gave **storage 0.3 / 10 GB (3%)** and **bandwidth 4.5 / 10 GB (45%)** against 197.6 MB of tracked objects. The entry now carries those numbers plus the part that is actually actionable: **bandwidth, not storage, is the metric to watch**, and its cost is dominated by the *infographics* (37 files averaging **4.44 MB**, so 164.3 MB for one full catalog view) rather than the *screenshots* added in v1.6.0 (83 files averaging **0.40 MB**, 33.3 MB) — five times cheaper in aggregate despite being more than twice as many files. It also records that bandwidth is billed on **downloads**, so `git lfs push` does not consume it. Finally it **retracts a scare**: v1.6.0's text warned that purging LFS objects from history does not reclaim GitHub-side storage and might need a support request — checked against the real bill, the ~234 MB of NotebookLM media purged in June is demonstrably **not** being billed (storage would read ≥0.43 GB if it were), so no request is needed. Documentation only; no behaviour changes.

## [v1.6.0] - 2026-07-18

_Zero-trust and supply chain, proven on a live cluster: Cloud Service Mesh, Binary Authorization signing on all four CI engines, the Backstage onboarding golden path, and a categorised catalog of the running platform._

_Both flags are **off by default** and were validated against a real cluster rather than reasoned about. Headlines: (1) **Cloud Service Mesh** (`serviceMesh.mode=cloud-service-mesh`) — managed Istio via the **standalone SKU**, not GKE Enterprise (that edition dissolved 2025-09); identity-based mutual **mTLS** + **L7 authZ** east-west, **mutually exclusive** with `gateway.backendTls` (a mesh supersedes that hop), enforced by a fail-fast gate in `lib/config.sh` **and** collapsed into one `intra_cluster_tls` dropdown in the GHA forms so the conflict is unexpressable. (2) **Binary Authorization** (`security.binaryAuthorization.enabled`) — GKE admits only images carrying a Cloud-KMS-signed **attestation** from this project's attestor, closing the [601](docs/601-DEVSECOPS.md) scan→deploy loop; `dryrun`(default)/`enforce`, **orthogonal** to the TLS/mesh axis, with one signing script shared by all four engines. **`enforce` proved live**: the same pipeline image is **denied** unattested and **admitted** once signed — only the attestation differs. (3) **Seven defects, and the live validation is what found them** — three would have hit any user who enabled the mesh (a sidecar CPU default that wedges every rolling update; the mesh silently stealing the OTel Java agent so every dashboard reads "No data"; STRICT mTLS rejecting the pipeline's own smoke test), one was the false-green check that hid the second, and one was a signing bug that would have failed **every** real build. (4) **Four-engine parity closed**: GitHub Actions gained the OTel Self-Heal step it never had — its RBAC had been granted for it all along. **(5) Image signing closed on all four CI engines** — Jenkins, Tekton, GitHub Actions/ARC and Argo Workflows now sign from one shared script, each enablement proven by a **real attestation** rather than a green step, and the `intra_cluster_tls × binauthz` orthogonality is backed by evidence in **all three** TLS states (`none`, `backend-tls`, `cloud-service-mesh`) instead of by argument. Getting there took three distinct per-engine wiring models and surfaced its own chain of gaps — the image digest, the pipeline service account, and finally `gcloud` simply not being on the ARC runner's PATH. **(6) The Backstage Scaffolder golden path** — *"Onboard an existing service"* renders the three files a service must be declared in across two repos and opens **one PR per repo**, never creating a repo (so no wider token scope). Three defects appeared only on the first real run while every static check stayed green, including a **fully green run that wrote invalid YAML** — the lesson being that a green run proves the actions executed, not that what they wrote parses. **(7) Backstage Monitoring made to survive backend-TLS** — a five-bug saga (a phantom datasource that crashlooped Grafana, a stale IAP audience after the backend service was recreated, a proxy target that lost its port, admin auth that was not idempotent to password rotation, and an alert sidecar reloading over http against an https listener), whose master lesson generalises: *every* `http://localhost` self-call inside a pod breaks the moment backend-TLS reaches it. **(8) `docs/screenshots/` — 83 captioned screenshots of the running platform**, Git-LFS-tracked and banded by the UI actually shown; the visual companion to the infographics catalog, and proof the platform runs end-to-end across all four engines and all four observability backends. **(9) Scheduling and first-run polish** — the prepull and `node-exporter` DaemonSets no longer sit `Unschedulable` on the tiny nodes GKE's *default* NAP provisions for an overflowing platform pod, and `jenkins.seedBuilds` stops the Jenkins CI/CD tab being an empty `notbuilt` job on a fresh cluster._

### Added
- **docs/505 § CI-engine integration — the *re-run* matrix**: a second comparison table beside the existing "what each tab shows" one, documenting whether you can **trigger a re-run from the Backstage CI/CD tab** per engine, and *why* the answer splits into two families — *native in-tab* (`jenkins` re-build, `githubactions` re-run) vs *observe-here-act-elsewhere* (`tekton` view-only → Tekton Dashboard, `argoworkflows` deep-link → Argo Server UI). The divergence is an **auth** divergence (Jenkins server-side API token · GitHub Actions the viewer's own OAuth · the two Kubernetes-native engines expose no in-tab write path), which is also why the re-build CSRF fix was Jenkins-specific. The `### Jenkins` prose now reflects the minted **API token** that replaced the seeded admin password.
- **`jenkins.seedBuilds` — pre-populate the Jenkins CI/CD tab on Day1** ([`06-seed-pipelines.sh`](scripts/06-seed-pipelines.sh)): after the seed job creates the microservices pipeline jobs, trigger **one build per job** (fire-and-forget) so the Jenkins CI/CD view — in **Backstage** and the Jenkins UI — isn't an empty `notbuilt` job on a fresh cluster. **Parity with `tekton.seedRuns` / `githubactions.seedRuns`**; default **true**, opt out with `jenkins.seedBuilds: false` / `JENKINS2026_JENKINS_SEED_BUILDS`. Found live: Backstage's Jenkins plugin **re-build** button crashes on a job with **no builds** (`Jenkins re-build has failed. Error: Cannot read properties of null (reading 'number')`) — the plugin can only *re-run an existing build*, never trigger the first one, and nothing had built the seeded jobs (only git-push webhooks or a manual *Build Now* do, unlike the seedRuns-pre-populated engines). The app itself deploys via GitOps regardless of this flag — this only exercises the pipeline and fills the build history. Reuses the existing crumb+cookie REST machinery (backend-TLS-safe); `microservices-k6-smoke` self-gates on service readiness so it is safe to trigger alongside the app builds.
- **Backstage Scaffolder — the "Onboard an existing service" golden path** ([docs/505 § Scaffolder](docs/505-BACKSTAGE.md)): the first template, reachable from **Sidebar → Create**. Adding a service here was never about writing code — it is that **one service must be declared in three files across two repos** (`jenkins/pipelines/seed/services.yaml`, the registry all four CI engines read; `backstage/catalog/services.yaml`, the entity and its eight annotations; and gitops-config's `values-stable.yaml`), where a single wrong annotation breaks a tab silently. The template renders all three from one form and opens **one PR per repo**. It **never creates a repo**, so it needs no extra token scope — it reuses the platform's existing `integrations.github` PAT — and it targets **`develop`, never `main`**, because this repo's `gitflow-guard` would reject the latter. Editing files it did not write is handled by a single ~40-line custom action, `j2026:file:append`: all three targets are **append-only by construction** (`services:` is the last key of both YAMLs, the catalog is a multi-document stream), so a new service is a pure append with no YAML parse, no merge, and no reformatting — the comments carrying those files' design rationale survive intact. Because that property is load-bearing but invisible, the action asserts an **anchor** and fails loudly if the files' shape ever changes, rather than appending into the wrong place. Merged PRs need **no image rebuild**: the catalog is read from git by URL. **The Roadmap's stated blocker for shipping any of this — "the Scaffolder drags the `isolated-vm` native toolchain (python3/g++) into the image" — was re-tested rather than assumed, and no longer holds:** `isolated-vm` 6.x ships `prebuilds/linux-x64`, verified loading inside the real `node:24-trixie-slim` base image with no compiler present, so the Dockerfile stays slim. **Three defects only the first real RUN could find**, every static check green throughout (`tsc`, the image build, the pod starting, the plugin registering its actions, the form rendering and validating): **(a)** `kind: Template` in `catalog.rules` is necessary but **not sufficient** — validating it needs `@backstage/plugin-catalog-backend-module-scaffolder-entity-model`, a package *separate* from `plugin-scaffolder-backend` and not pulled in by it; without it the catalog **read** the template and dropped it at `warn` level, leaving the Create page empty with no error (`a72633c`). **(b)** `isolated-vm` on Node ≥20 needs `NODE_OPTIONS=--no-node-snapshot` or every template run dies at its first step — now `ENV` in the image and mirrored in the backend's start script for local `yarn dev` (`cc9403a`). **(c)** The run then went **fully green while writing broken YAML** (`bf21cde`): a plain block scalar (`|`) auto-detects its indentation from the first non-empty line and **strips** it, so entries landed at column 0 while the existing ones sit at 2 — `services.yaml` came out **invalid** (that file is the registry all four engines read, so merging it would have broken every one) and, worse because it still *parses*, the gitops values put the service at **top level instead of under `services`**, a silent no-op ArgoCD would render nothing for. Fixed with the explicit `|2` indicator and verified by simulating the append against the real files and **re-parsing the output**. Finally (`674b411`), onboarding an already-registered name produced a valid file with a **duplicate** entry — YAML accepts that happily, and the seed job would have created two jobs under one name where the second **shadows** the first, silently re-pointing an existing service's pipeline at the new repo; an `absent` input (compared per-line, never as a substring, so `name: gateway` cannot match `name: gateway-v2`) now refuses. The lesson the template's own comments carry: **a green run proves the actions executed, not that what they wrote is correct — only re-parsing the emitted YAML does.** (Generating a *new* service from a skeleton remains the next increment: it needs a template repo and repo-creation scope, and is worth doing when a third service makes golden paths real.)
- **Cloud Service Mesh (CSM), standalone SKU — opt-in `serviceMesh.mode`** ([docs/506](docs/506-SERVICE-MESH.md), [ADR 0001](docs/adr/0001-intra-cluster-security-and-supply-chain-admission.md)): `none` (default) | `cloud-service-mesh`. `terraform/gke/security.tf` registers the cluster to a **Fleet** and enables the **`servicemesh` feature** (`MANAGEMENT_AUTOMATIC`) — Google runs `istiod` + the **Mesh CA**, so there is no self-managed control plane to patch; `scripts/08.85-service-mesh.sh` does the in-cluster half (namespace injection label, `PeerAuthentication` STRICT, the ingress-edge + non-mesh-caller port exceptions, an ordered restart of only the app deployments), with a **symmetric retire** so flipping the flag off converges with no manual `kubectl`. **Mutually exclusive with `gateway.backendTls`** (docs/504): a mesh's Mesh CA owns the LB→pod hop *and* adds east-west identity, so running both is redundant — `lib/config.sh` **fails fast** if both are set, and the `Day1`/`Day2.redeploy.*` forms expose a single `intra_cluster_tls` choice (`none`/`backend-tls`/`cloud-service-mesh`) that makes the conflict impossible to express. Docs carry the **why-CSM** and **why-NOT** decision matrices (Istio self-managed sidecar *and* ambient, Traefik, Linkerd, Cilium, other meshes), the *"Dataplane V2 **is** Google-managed Cilium"* explainer, and the **per-mesh-client** cost model (~$0.50/client/month — cents for an ephemeral PoC; node/vCPU count is irrelevant to the mesh bill).
- **Binary Authorization — opt-in `security.binaryAuthorization.enabled` + `enforcementMode`** ([docs/507](docs/507-BINARY-AUTHORIZATION.md)): `terraform/gke/security.tf` creates the Cloud KMS asymmetric key, the Container Analysis note, the attestor, the project singleton policy (allowlisting GKE system images) and the `jenkins-2026-binauthz-signer` GSA with Workload-Identity bindings for **all four engines' build KSAs**; the cluster consults the policy at admission. `resources/sign-and-attest-image.sh` is the **single signing source** for the four engines (the `patch-app-source.sh` pattern) — it resolves the immutable **digest** (never a mutable tag) and no-ops unless `BINAUTHZ_ENABLED=true`, so an inactive cluster runs the step as a harmless no-op. **`dryrun`** (audit-log, default) / **`enforce`** (block), **orthogonal** to the TLS/mesh axis. Jenkins — the default engine — is wired end-to-end: `04-jenkins.sh` patches the flag into `jenkins-credentials` → `containerEnv` → JCasC `globalNodeProperties` → the agent, and threads the signer-GSA annotation as a **chart helm-parameter** (the `jenkins` KSA is ArgoCD-managed, so a raw `kubectl annotate` would drift).

### Fixed
- **The three PR-guard badges rendered `no status` (grey) instead of green** ([`README.md`](README.md)): the **Gitflow Guard**, **Terraform validate** and **Mermaid validate** badges were pinned with `?branch=main`, but all three workflows trigger **only** on `pull_request` — and GitHub attributes a `pull_request` run to the PR's **head** branch (`develop`), never to its base. So the API had **zero** runs matching `branch=main` for any of them and served the grey *no status* badge, even though every recent run is `success`. Dropping the `?branch=main` pin makes all three report **passing** (verified by fetching the three SVGs before and after the change). The obvious alternative — adding `push: branches: [main]` to the workflows so they *do* run on main — was **rejected**: it is semantically wrong for `gitflow-guard`, which exists to assert that a PR originates from `develop` and has no PR to inspect on a `push` event, and it is pure duplicated work for the other two, which already ran on the very PR that produced the commit. Nothing was broken; the badge was asking GitHub a question that, by design, can never have an answer.

## [v1.0.0] - 2026-07-02

_First stable release._ Promotes **jenkins-2026** from its `v0.x` rapid-development line (v0.1.0–v0.29.0, now a frozen historical record — its last increment consolidated in [v0.29.0](#v0290---2026-07-02)) to a stable **1.0** baseline. No functional change versus v0.29.0; this release declares the reference PoC feature-complete and adopts the 1.0 versioning policy in [`RELEASING.md`](RELEASING.md) (from here: minor = features, patch = fixes, major = breaking).

### What v1.0.0 delivers

The complete, working reference platform — a Jenkins-on-Kubernetes CI/CD PoC with full OpenTelemetry observability, pluggable along three axes:

- **CI engine** (`ci.engine`) — **four interchangeable engines** running the same 10-stage pipeline from one `jenkins/pipelines/services.yaml`: **Jenkins** (default), **Tekton**, **GitHub Actions/ARC**, **Argo Workflows**. Switching engines fully retires the previous one.
- **Observability backend** (`observability.mode`) — **four backends** with correlated traces/metrics/logs, dashboards + alerts provisioned as code, and synthetic Faro RUM: in-cluster **OSS** (Grafana/Loki/Tempo/Prometheus), **Grafana Cloud**, **Azure Managed Grafana**, **Amazon Managed Grafana**.
- **Secrets backend** (`secrets.backend`) — `imperative` (default) or `eso` (GCP Secret Manager + External Secrets Operator, keyless WIF).
- **Platform** — GitOps via **ArgoCD** (app-of-apps), **HA CloudNative-PG** Postgres, **Gateway API + Google IAP** ingress, **Dataplane V2 + WireGuard** networking, elastic **Spot Node Auto-Provisioning**.
- **Lifecycle** — a clean, idempotent **Day0 → Day1 → Day2 → Decom** flow; one-command bootstrap; **rebuild-safe** by design (`docs/104-REBUILD_SAFETY.md`).
- **Docs & release process** — ~20 numbered deep-dive docs, plus this scannable changelog + [`RELEASING.md`](RELEASING.md) convention with [`scripts/cut-release.sh`](scripts/cut-release.sh).

### Changed
- **Adopted a stable `v1.0.0` baseline and 1.0 versioning semantics.** The `v0.x` line is frozen as pre-release history (not renumbered — preserving version immutability); [`RELEASING.md`](RELEASING.md) now documents minor/patch/major from 1.0 on.
